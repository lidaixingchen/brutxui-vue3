## 一、高严重度

### H-1. transformHead 对英文首页生成错误的 canonical / hreflang / JSON-LD URL
- **文件**：[apps/docs/.vitepress/config.ts](file:///e:/project/brutxui-vue3/apps/docs/.vitepress/config.ts#L137-L154)
- **类型**：逻辑 / SEO 配置
- **现象**：英文首页 `en/index.md` 经 `replace(/\.md$/, '')` → `en/index` → `replace(/\/index$/, '')` → `en`。随后 `basePath = pageUrl.replace(/^en\//, '')` 的正则 `^en\/` 要求 `en/` 前缀，对裸字符串 `'en'` 不匹配，`basePath` 仍为 `'en'`。
- **结果**：
  - `canonicalUrl = ${baseUrl}/en/en`（应为 `${baseUrl}/en/`）
  - hreflang `zh-CN` = `${baseUrl}/en`（应为 `${baseUrl}/`）
  - hreflang `en` = `${baseUrl}/en/en`（应为 `${baseUrl}/en/`）
  - hreflang `x-default` = `${baseUrl}/en`（应为 `${baseUrl}/`）
  - JSON-LD `url` 同样错误
- **影响**：英文首页 SEO 严重受损，搜索引擎可能判定为重复页面或死链；社交分享与结构化数据均指向不存在的 `/en/en` 路径。
- **建议修复**：将正则改为 `/^en\/?/`，使 `'en'` → `''`、`'en/guide/foo'` → `'guide/foo'`；或在 `basePath` 计算后对 `pageUrl === 'en'` 单独置空。

## 二、中严重度

### M-1. CHANGELOG 生成在缺少 `\n## [` 标记时静默丢失全部历史
- **文件**：[scripts/release/generate-changelog.mjs](file:///e:/project/brutxui-vue3/scripts/release/generate-changelog.mjs#L184-L187)
- **类型**：逻辑 / 数据丢失
- **现象**：`firstEntryIndex = existing.indexOf('\n## [')`。若返回 `-1`（如 CHANGELOG 被手动改成首行直接 `## [`、或采用其他标题格式），代码进入 `contentWithoutHeader = ''` 分支，写入 `defaultHeader + newEntry + '\n' + ''` 完全覆盖原文件。
- **影响**：用户手动编辑 CHANGELOG 后，下一次 `pnpm changelog` 会抹掉所有历史条目，且无错误提示。
- **建议修复**：`firstEntryIndex === -1` 时应保留 `existing` 作为尾部内容（`header + newEntry + '\n' + existing`），或直接 fail 并提示用户检查格式。

### M-2. 代码生成器在部分导出已存在时跳过全部新导出
- **文件**：[scripts/generate.ts](file:///e:/project/brutxui-vue3/scripts/generate.ts#L619-L626)
- **类型**：逻辑
- **现象**：`updateIndexFile` 中 `duplicateExports = exports.filter(exp => content.includes(exp))`，只要任一导出已存在就 `return true`，其余未存在的导出不会被追加。
- **影响**：重新生成同名组件或部分文件已存在时，`index.ts` 缺失导出但脚本仍报成功，导致组件未被导出、构建静默失败。
- **建议修复**：过滤出 `exports` 中不在 `content` 的子集，仅追加该子集；全部已存在时再走提示分支。

## 四、模式汇总

### 1. 定时器无清理（5 个文件）
`setTimeout` / `setInterval` 在事件回调中创建但未在 `onUnmounted` 清理，是本批次最普遍的模式。全量 demo 文件中 `onUnmounted` / `onBeforeUnmount` 出现次数为 0，说明这是全局性疏漏而非个别遗漏。文档站场景下危害有限，但作为组件库示例代码应树立正确范式。

涉及文件：UploadDemo、LoadingDemo、CounterDemo、ComboboxDemo。

### 2. i18n 不统一（3 个文件）
存在三种 i18n 实现方式并存：
- `useI18n()` 响应式（标准，部分组件使用）
- 模板内联三元 `locale === 'en' ? '...' : '...'`（ComponentCatalog）
- 完全硬编码中文（ThemePlayground、TranslationBanner）

导致 `i18n.ts` 中部分 key 成为死代码，且部分英文页面显示中文。

### 3. CI 加固缺失
4 个工作流普遍缺少：concurrency 控制（3/4 缺失）、最小化 permissions（2/4 缺失）、SHA pin（4/4 全部使用 tag）。publish.yml 作为发布关键工作流，三项加固均缺失，风险最高。

### 4. 调试代码残留（3 个文件）
TransferDemo、RateDemo、PopconfirmDemo 均在事件处理函数中遗留 `console.log`，属提交前未清理的调试代码。

### 5. SSR 安全良好
全量扫描未发现 setup 顶层直接访问 `window`/`document`/`localStorage` 等浏览器 API 的情况。所有浏览器 API 调用（`window.prompt`、`URL.createObjectURL` 等）均在事件处理函数内，VitePress SSR 构建不会崩溃。

### 6. 无 XSS / 响应式陷阱
全量扫描未发现 `v-html` 注入、`reactive()` 整体替换、`computed()` 副作用、`JSON.parse` 无 try-catch、`parseInt` 无基数等问题。
