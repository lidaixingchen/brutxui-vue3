## 一、新发现

### N-1. HomeCodePreview.vue 示例代码使用不存在的导出名 `BButton`（中）

- **文件**：[apps/docs/.vitepress/theme/components/HomeCodePreview.vue#L36,L40,L45,L49](file:///e:/project/brutxui-vue3/apps/docs/.vitepress/theme/components/HomeCodePreview.vue#L33-L53)
- **类型**：文档示例错误
- **现象**：首页"快速开始"代码示例的 `usageCode` computed 中两段模板（中/英）均写：

  ```vue
  import { BButton } from '@/components/ui/button'

  <template>
    <BButton variant="primary" size="lg">
      Neo-Brutalism Button
    </BButton>
  </template>
  ```

  但 [packages/ui/src/components/button/index.ts#L7](file:///e:/project/brutxui-vue3/packages/ui/src/components/button/index.ts#L7) 实际导出的命名为 `Button`（`export { default as Button } from './Button.vue'`），项目其他所有文档/示例均使用 `Button`。Grep 验证：`BButton` 字样仅出现在本文件，**整个仓库别处无任何引用**，确认是孤立的错误命名。
- **影响**：用户复制首页示例代码到本地工程后 `import { BButton }` 会得到 `undefined`，模板渲染时报 `Failed to resolve component: BButton`，**首次接入即失败**。首页作为门面，给新用户造成"库不可用"的负面印象。
- **建议修复**：将 6 处 `BButton` 改为 `Button`，或与 `HomeComponentShowcase.vue` 的导入方式统一为 `import { Button } from 'brutx-ui-vue'`。
- **回归来源**：从旧报告 L-11（a11y 修复）引入 a11y 改造时同时重写了示例代码，命名错误在此期间被引入。提示后续修复 a11y 类问题时需同步复核示例代码片段。

### N-2. bench.yml 第三方 Action 未按 SHA 锁定且持 PR 评论写权限（中）

- **文件**：[.github/workflows/bench.yml#L73,L81](file:///e:/project/brutxui-vue3/.github/workflows/bench.yml#L72-L86)
- **类型**：供应链 / CI 权限
- **现象**：`pr-compare` job 末尾使用两个第三方 Action：
  - `peter-evans/find-comment@v3`（L73）
  - `peter-evans/create-or-update-comment@v4`（L81）

  均以 tag 引用，未按 commit SHA 锁定。同时 [bench.yml#L11-L13](file:///e:/project/brutxui-vue3/.github/workflows/bench.yml#L11-L13) 声明 `permissions: { contents: read, pull-requests: write }`，向这两个第三方 Action 显式授予了 PR 评论写权限。
- **影响**：
  - **供应链风险**：`peter-evans/*` 仓库被入侵或 tag 被移动后，攻击者可在任意 PR 下注入评论（含钓鱼链接）。
  - **权限放大**：相比 `actions/*` 官方 Action，第三方 Action 持 `pull-requests: write` 的风险面更大；该 Action 还接收 `github.event.pull_request.number` 作为参数。
  - 与旧报告 L-17 性质相同，但因引入新工作流而被记为新发现。
- **建议修复**：
  1. 将 `peter-evans/find-comment@v3` 与 `peter-evans/create-or-update-comment@v4` 改为对应 commit SHA + 校验和，例如 `peter-evans/find-comment@<sha>`。
  2. 评估是否可改用 GitHub 官方 `actions/github-script` + REST API 自行实现评论 upsert，减少对第三方 Action 的依赖。
  3. 若坚持使用第三方 Action，建议在 workflow 顶部 `concurrency` 之外再添加 `environment` 隔离。

### N-3. scan-phantom-deps.mjs 的 DECLARED 映射为硬编码，新增依赖易遗漏同步（低）

- **文件**：[scripts/scan-phantom-deps.mjs#L18-L52](file:///e:/project/brutxui-vue3/scripts/scan-phantom-deps.mjs#L15-L52)
- **类型**：可维护性 / 配置漂移
- **现象**：`DECLARED` 对象为硬编码的 `Record<pkgName, Set<string>>`，L16-L17 注释明确写道：

  > Keep in sync with the actual package.json files; the script does NOT auto-read them so the diff is explicit when a new phantom appears.

  当前覆盖 5 个包（`brutx-ui-vue`、`brutx-vue`、`brutx-registry-vue`、`brutx-shared-vue`、`docs`）共 ~70 条依赖项。
- **影响**：
  - 当某包新增依赖时，开发者需手动同步本文件；忘记同步后，新依赖的导入会被误报为 phantom，CI 噪声增加，甚至让开发者养成"忽略 phantom 报警"的习惯。
  - 当某包移除依赖时反向问题更严重：脚本不再报警，真正的 phantom 反而被掩盖（因为仍在硬编码白名单中）。
- **建议修复**：改为运行时读取各 `package.json` 的 `dependencies + devDependencies`：

  ```js
  import { readFileSync } from 'node:fs'
  function loadDeclared(pkgDir) {
      const pkg = JSON.parse(readFileSync(join(ROOT, pkgDir, 'package.json'), 'utf8'))
      return new Set([...Object.keys(pkg.dependencies ?? {}), ...Object.keys(pkg.devDependencies ?? {})])
  }
  ```

  若仍希望保留显式 diff 语义，可改为"读取 package.json 后与硬编码 baseline 对比，diff 时发出警告"的双层模式。

### N-4. bench-diff.mjs 在读取基准文件失败时硬退出，CI 流程中断（低）

- **文件**：[scripts/bench-diff.mjs#L25-L34](file:///e:/project/brutxui-vue3/scripts/bench-diff.mjs#L25-L34)
- **类型**：错误处理 / CI 韧性
- **现象**：`loadBenchResult` 函数在 `readFileSync` 抛错或 `JSON.parse` 失败时直接 `console.error` + `process.exit(1)`：

  ```js
  function loadBenchResult(file) {
      let raw
      try { raw = readFileSync(file, 'utf8') } catch (e) {
          console.error(`Failed to read ${file}: ${e.message}`)
          process.exit(1)
      }
      // ...
  }
  ```

  调用方 [bench.yml#L70](file:///e:/project/brutxui-vue3/.github/workflows/bench.yml#L69-L70) 是 `node scripts/bench-diff.mjs ... > bench-report.md`，一旦失败 `bench-report.md` 为空文件，后续 `Create or update comment` 步骤会用空 body 发评论。
- **影响**：
  - bench baseline artifact 上传失败（如 baseline job `continue-on-error` 吞错时）会直接让 PR 评论 job 失败。
  - 用户体验上得到的是一个空评论或失败红叉，无法快速判断是"性能回归"还是"基础设施故障"。
- **建议修复**：将 `process.exit(1)` 改为抛出含上下文的自定义 Error，由顶层 `main()` 捕获后输出结构化错误报告到 `bench-report.md`，让 PR 评论至少能显示"基准数据缺失，请重试"。

## 四、模式汇总

### 1. SHA pin 缺失持续未治理（系统性问题）

5 个工作流中所有 `uses:` 引用（共 37 处）均为 `name@vN` 形式。`peter-evans/*` 第三方 Action 的引入让该问题在 P2 改造后**风险等级上升**——从"仅官方 Action 被移动 tag"扩展到"第三方 Action 被入侵即可写 PR 评论"。

### 2. P2 改造引入的脚本韧性不足（新趋势）

P2 引入的三个新脚本（`check-i18n.mjs`、`scan-phantom-deps.mjs`、`bench-diff.mjs`）均存在错误处理范式单一的问题：
- `scan-phantom-deps.mjs`：依赖硬编码白名单，与 `package.json` 双源同步风险（N-3）。
- `bench-diff.mjs`：`process.exit(1)` 硬退出，CI 评论 job 失败时无降级（N-4）。
- `check-i18n.mjs`：CI 中以 `continue-on-error: true` 接入，缺少失败原因的结构化输出。

共同模式：脚本能跑通正常路径，但异常路径下用户体验差。

### 3. i18n 已统一，无新问题

P2 改造后所有文档站组件（含 `ThemePlayground`、`TranslationBanner`、`ComponentCatalog`、`HomeCodePreview`、`HomeComponentShowcase`、`HomeStats` 等）均使用 `useI18n()` 响应式 `t()`。无新增硬编码中文/英文文案。

### 4. 内存泄漏问题已根除

所有 demo 文件（`UploadDemo`、`LoadingDemo`、`CounterDemo`、`ComboboxDemo`、`CopyButton`、`ThemePlayground`）的 `setTimeout`/`setInterval`/Blob URL 均已在 `onUnmounted`/`onBeforeUnmount` 中清理。范式统一。

### 5. SSR 安全与 a11y 良好

- 未发现 setup 顶层直接访问浏览器 API 的情况。
- `InstallationTabs`、`HomeCodePreview` 的 roving tabindex 与 `aria-controls`/`aria-labelledby` 关系正确。

### 6. 无 XSS / 响应式陷阱

未发现 `v-html` 注入、`reactive()` 整体替换、`computed()` 副作用、`JSON.parse` 无 try-catch 等问题。
